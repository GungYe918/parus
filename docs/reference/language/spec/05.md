## 3.4 `-ffreestanding` / `-fno-std` 경계 정의

참고:

1. bundle/module/import/export 가시성 규칙 정본은 `docs/reference/language/spec/30_BUNDLE_MODULE_RESOLUTION.md`를 따른다.
2. `import foo::bar` / `import ::foo::bar` 절대 표기 규칙도 위 정본 문서를 따른다.
3. 본 문서는 런타임/표준 라이브러리 경계 규칙만 다룬다.

### 3.4.1 두 모드의 목적

* `-ffreestanding` : OS/호스트 환경을 가정하지 않는다.

  * 표준 라이브러리는 “freestanding 프로파일”로 링크 가능(선택)
  * 원자/스핀락/기본 핸들 같은 최소 런타임 도구를 제공 가능
* `-fno-std` : 표준 라이브러리 **전체 미사용**.

  * `core`(컴파일러 내장 최소 모듈)만 사용 가능
  * 런타임 의존 기능은 전부 금지

---

### 3.4.2 예외 기능과의 경계(v0 고정)

예외 메커니즘(`?`, `throw`, `try...catch`, `try expr`)은 언어 코어 기능이다.
다만 코어가 보장하는 것은 **Recoverable non-unwind 채널**이며,
언와인딩 런타임은 v0 코어 필수가 아니다.

채널 분리 원칙:

1. Recoverable(`throw`)는 값 기반 채널이다(비-언와인딩).
   내부 ABI에서는 hidden `exc_ctx*`와 thread-local root exception context를 사용한다.
2. Unrecoverable(panic/foreign fault)는 ordinary `try...catch`의 포획 대상이 아닌 fatal 채널이다.
3. panic 기본 정책은 `abort/trap`이다.
4. foreign C++/언와인드 브리지는 hosted/interop 후속 레이어 과제다.

* `-fno-std` 에서는:

  * 표준 라이브러리 헬퍼 없이 코어 예외 문법만 사용한다.
  * `throw` payload는 코어 허용 타입(`enum`, `struct` 값)으로 제한한다.
  * non-`?` 함수는 `throw`/`try...catch`를 사용할 수 없고, `try expr`만 허용한다.
  * ordinary `?` 함수의 정상 경로는 hidden `exc_ctx*` 전달과 local check만 부담한다.
  * 기본 프로필은 언와인딩 런타임을 링크하지 않는다.
* `-ffreestanding` 에서는:

  * 위 코어 규칙을 동일하게 적용한다.
  * 표준 라이브러리 의존 없이도 Recoverable 채널이 동작해야 한다.
  * panic은 기본적으로 `abort/trap`로 처리한다.

이 경계 덕분에:

* 커널/펌웨어/부트로더 같은 환경에서도 코어 예외 모델을 동일하게 사용할 수 있다.
* 표준 라이브러리 부재 여부와 무관하게 언어 의미론이 일관된다.
