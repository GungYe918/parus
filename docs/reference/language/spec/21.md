## 10. Type Declarations (struct/enum/proto/class)

문서 상태: `Normative (Proto v1: Constraint-Only)`

### 10.1 struct

1. `struct`는 데이터 선언이다.
2. `struct`는 `proto` 구현 대상으로 선언될 수 있다.
3. `struct` 본문은 계속 field-only다.
4. 다만 `struct : Proto` 충족 검사는 default `acts for T`를 함께 본다.
   즉 proto requirement 함수/associated type은 struct 본문이 아니라 default acts가 채울 수 있다.

예시:

```parus
struct UserId : Marker {
  u64 value;
};
```

### 10.2 proto (constraint-only)

1. `proto`는 제약/계약 선언 전용이다.
2. `proto`는 런타임 엔티티를 생성하지 않는다(완전 소거).
3. `proto` 멤버는 `require`/`provide`로 시작해야 한다.
4. `proto` 본문은 아래 항목만 허용한다.
`require struct(Path);`
`require enum(Path);`
`require class(Path);`
`require actor(Path);`
`require acts(Path);`
`require type Name;`
`require foo(...) -> Ret;` (`def` 금지)
`provide def foo(...) -> Ret { ... }` (본문 필수)
`provide const NAME: T = expr;` (정적/불변 상수)
5. `proto ... with require(...)` 꼬리 문법은 제거되었다.
6. `proto A: B, C`에서 `:` 뒤 대상은 `proto`만 허용한다.
7. `require` 항목에는 `proto(...)`를 허용하지 않는다.
8. `struct/enum/proto/class/acts/actor`처럼 블록 본문을 가지는 선언은 선언 끝 `;`를 선택적으로 생략할 수 있다.

기본 문법:

```parus
proto Equatable {
  require eq(lhs: Self, rhs: Self) -> bool;
};
```

### 10.3 self / Self 규칙

1. `Self`는 proto 구현체 타입 플레이스홀더다.
2. `Self`는 타입 위치에서 사용한다.
3. `Self::Assoc`는 proto의 required associated type을 가리킨다.
4. `self`는 class/actor/acts 메서드 리시버에서 사용한다.
5. proto `require/provide` 함수 시그니처에서 `self` 파라미터 선언은 금지한다.
6. 시그니처 매칭 시 `Self`/`&Self`/`Self::Assoc`는 구현체 + acts binding 기준으로 정규화한다.

### 10.4 proto require/provide 적용 규칙 (v1)

1. `require struct/enum/class/actor/acts(Path);`는 심볼 존재/가시성/종류 일치를 강제한다.
2. 같은 파일 선언은 허용한다.
3. 같은 모듈 타 파일 선언은 `export`된 심볼만 허용한다.
4. 모듈 밖 심볼은 `import/alias`로 반입되고 `export`된 경우만 허용한다.
5. `require type Name;`는 구현체가 binding 해야 하는 associated type slot을 추가한다.
6. `require foo(...) -> T;`는 구현체가 충족해야 하는 함수 계약을 추가한다.
7. `provide def`는 기본 구현을 제공하며, 상위 proto에서 승계된 `require` 함수를 시그니처로 충족할 수 있다.
8. default `acts for T`는 proto 충족 검사에 참여한다.
   * 함수 requirement는 default acts method로 충족할 수 있다.
   * associated type requirement는 acts header witness(`<... is Assoc>`)로 충족할 수 있다.
9. `effective required fn set = closure(require fn) - closure(provide fn)`으로 계산한다.
10. 구현 적용 시 effective required fn과 required associated type set이 모두 충족되어야 한다.
11. `struct/enum`도 default acts를 통해 non-empty proto requirement를 충족할 수 있다.
12. `provide const`는 인스턴스 저장공간을 만들지 않는 정적(read-only) 상수이며, 컴파일타임 평가 가능한 초기화식이 필요하다.
13. `type -> proto` + `proto -> type/acts` 경로를 포함한 proto/type/acts 강결합 순환은 컴파일 오류다.

### 10.4.a require(expr); 문장 (컴파일타임 단정)

1. `require(expr);`는 함수/블록 문맥에서 사용하는 컴파일타임 단정 문장이다.
2. `expr`는 v0에서 단순 bool 폴딩(`true/false/not/and/or/==/!=`)으로 접혀야 한다.
3. `expr`가 `false`로 접히면 컴파일 오류가 발생한다.

예시:

```parus
def main() -> i32 {
  require(true and not false);
  return 0i32;
}
```

### 10.4.b compiler 지시문 (`$[]` / `$![]`)

1. `$[]`는 호출식 기반 단일 item 지시문이다. 문법은 `$[Callee(...)] <item-decl>`이다.
2. `$[If(...)]`는 컴파일러 builtin이다. 라이브러리/유저 선언으로 대체할 수 없다.
3. `$[if(...)]` 소문자 레거시 문법은 제거되었고, 사용 시 오류가 발생한다.
4. `$[]`의 사용자 확장은 `inst` 선언으로 제공한다. `inst Foo(...) { ... }`는 컴파일타임 bool 판정 단위다.
5. `inst`는 `-> bool` 표기를 사용하지 않으며 암묵적으로 bool을 반환한다.
6. `inst` v1 인터프리터 허용 범위는 `let`, `if/else`, `return`과 단순 식(bool/int/string literal, ident, `not/and/or`, `==/!=`, 다른 inst 호출)이다.
7. `inst` 평가식/본문에서 `&`, `~`, `~&`는 금지된다.
8. `$![]`는 두 형태를 지원한다.
   1. 대입형: `$![key.path = target::path];`
   2. 태그형: `$![tag.path];`
9. `$![]`는 item-attached form도 지원한다.
   1. attached `Impl::*` binding은 현재 function item에만 허용한다
   2. 예: `$![Impl::SpinLoop] export def spin_loop() -> void;`
10. `$![Impl::Core];`는 파일 헤더 전용 표식이다.
    1. top-level 첫 item 위치에서만 허용한다.
    2. 파일당 1회만 허용한다.
    3. 유효하면 parser 내부 `core-file mode`를 활성화한다.
11. attached `Impl::*` binding의 의미는 body 유무로 나뉜다.
    1. recognized `Impl::*` + body 없음 => compiler-owned impl point
    2. recognized `Impl::*` + body 있음 => library-owned ordinary function
    3. unknown `Impl::*` + body 있음 => metadata-only ordinary function
12. body 없는 `def ...;`는 recognized compiler-owned `Impl::*` binding에만 허용한다.
13. recognized compiler-owned `Impl::*` surface는 bundle `core` + 같은 파일의 `$![Impl::Core];`를 요구한다.
14. `v1`에서 `$![]`는 파싱/AST 보존 중심이며, attached `Impl::*` policy와 core gate는 후속 tyck 단계에서 확정한다.
15. `proto`는 제약/공급 모델이고, `$[]/$![]`는 compiler directive 모델이다. 둘은 역할이 겹치지 않는다.

### 10.5 class

1. `class`은 구현체 타입이다.
2. `class Name : ProtoA, ProtoB` 형태로 proto 구현을 명시한다.
3. `require` 함수 항목이 구현 강제 대상이다.
4. `provide def`로 충족되는 `require` 함수는 class에서 생략 가능하다.
5. lifecycle 멤버는 `init(...) { ... }`, `deinit() { ... }`, `init() = default;`, `deinit() = default;`를 지원한다.
6. `init()/deinit()`은 `->` 반환 타입을 선언하지 않는다.
7. `init()/deinit()`에서는 사용자 소스에서 `self` 리시버를 선언하지 않는다.
8. lifecycle 함수 내부 ABI는 hidden receiver `self: &mut Self`를 자동 주입한다.
9. `init()/deinit() = default;`는 무인자 사용자 시그니처만 허용한다.
10. `A(...)`는 class `A`의 `init(...)` 오버로드를 호출하는 생성식이다(v0: tmp storage 생성 후 `init(..., self=&mut tmp)` 호출).
11. `init/deinit` 직접 호출(`a.init(...)`, `a.deinit()`, `A::init(...)`, `A::deinit(...)`)은 금지된다.
12. dot 호출 `a.m(...)`은 첫 파라미터가 `self`인 메서드만 허용한다.
13. class/proto 멤버 경로 호출 `A::m(...)`, `Proto::m(...)`는 제거된다.
14. proto `provide` 멤버 접근/호출은 `->`만 사용한다(`a->f()`, `a->Proto.f()`, `a->CONST`).
15. `.`는 concrete 필드/메서드 전용이며 proto 멤버를 탐색하지 않는다.
16. `->`는 proto 멤버 전용이며 concrete 필드/메서드를 탐색하지 않는다.
17. `a->name` 후보가 여러 proto에서 충돌하면 `a->Proto.name` 한정자를 강제한다.
18. class 인스턴스 필드 선언은 `name: Type;`만 허용한다.
19. class 내부 `let/set/mut` 멤버 선언은 제거되며 하위호환을 제공하지 않는다.
20. 인스턴스 필드 초기화식(`name: Type = expr;`)은 허용하지 않는다.
21. class 정적 멤버는 `static def ...`와 `static name: Type = expr;`를 지원한다.
22. `static mut`은 v0에서 지원하지 않는다.
23. class 본문은 contextual visibility label `public:` / `private:`를 지원한다.
24. 기본 visibility는 `public`이다.
25. `private` 멤버는 선언한 class 본문에서 정의된 `init/deinit/method/static member` 안에서만 접근할 수 있다.
26. `acts for SomeClass`는 class 바깥 extension이므로 private 멤버 접근 권한을 얻지 않는다.
27. nested drop은 v0에서 지원한다. 정리 순서는 `user_deinit(self)` 실행 후 필드 선언 역순(drop reverse)이다.
28. `~` 이동은 v0에서 루트 식별자 이동(`~x`)만 허용하며 subplace move-out(`~self.x`, `~arr[i]`)은 금지한다.

예시:

```parus
proto Widget {
  provide def id() -> i32 {
    return 1i32;
  }
};

class Button : Widget {
  value: i32;
};
```

```parus
class User {
  init() = default;
  deinit() = default;

  def id(self) -> i32 {
    return 7i32;
  }
};

def main() -> i32 {
  set u = User();
  return u->id();
}
```

### 10.6 enum

1. `enum`은 생명주기(`init/deinit`)가 없는 값 타입이다.
2. 선언은 `case` + `,` 문법을 사용한다.
3. payload 선언은 `case V(name: T, ...)` 형식을 사용한다.
4. 생성은 `Type::Variant(...)`로만 허용한다.
5. payload 접근은 `switch` 패턴 바인딩 문맥에서만 허용한다(`e.value` 직접 dot 접근 금지).
6. nullable 결합은 `E?` + `null`을 사용한다.
7. `layout(c)` enum에서는 payload variant를 금지한다.
8. discriminant(`= n`)는 `layout(c)` + tag-only enum에서만 허용한다.
9. enum은 `acts` 부착을 허용한다.
10. enum의 proto 결합도 default acts를 통한 requirement 충족을 허용한다.

예시:

```parus
enum Token<T> {
  case Eof,
  case Int(value: i32),
  case Named(name: string, value: T),
};

def main() -> i32 {
  set t = Token<i32>::Named(name: "x", value: 1i32);
  switch (t) {
  case Token<i32>::Eof: { return 0i32; }
  case Token<i32>::Int(value: v): { return v; }
  case Token<i32>::Named(name: n, value: v): { return v; }
  }
  return -1i32;
}
```

### 10.7 dyn (문서 고정, 비구현)

1. 런타임 동적 디스패치/업캐스팅 경계는 `dyn`으로 명시한다.
2. 현재 라운드에서는 `dyn`을 파서/타입체커/IR에 구현하지 않는다.
3. ABI 고정 이후 별도 라운드에서 도입한다.
