# 29. Macro System (v0.1, Token Core Enabled)

문서 상태: `Draft`

이 문서는 Parus 매크로 시스템의 현재 구현 기준을 정리한다.  
정본 EBNF는 `/Users/gungye/workspace/Lang/gaupel/docs/reference/language/spec/28.md`를 기준으로 하며, 본 문서는 구현 정책/의미 규칙을 보강한다.

## 29.1 범위

1. 지원 문맥: `expr`, `stmt`, `item`, `type`, `token`.
2. 호출 문법: `$path(...)`.
3. 선언 문법: `macro name -> { with <kind> { ... } }`.
4. `with token`은 기본 활성화다(실험 플래그 없음).
5. compiler directive `$[...]`, `$![...]`는 매크로 시스템과 별개 계층이다.

## 29.2 호출/확장 규칙

1. 그룹 선택은 호출 문맥(`expr/stmt/item/type/token`) 기준으로 수행한다.
2. arm 선택은 선언 순서 first-match다.
3. 캡처 참조는 이름/positional(`$0`, `$1`, ...) 모두 허용한다.
4. variadic 캡처는 `$n...`, `$name...`를 허용한다.
5. 확장 결과는 arm의 `OutKind` 문맥으로 재파싱한다.
6. 매칭 실패/재파싱 실패는 fallback 없이 컴파일 오류다.

## 29.3 `with token` 지원 규칙

1. 패턴 요소:
`literal`, `capture`, `group`, `repeat($(...) * + ?, separator 지원)`.
2. fragment 캡처:
`tt/expr/stmt/item/type/path/ident/block`.
3. repeat 패턴:
`$( ... )*`, `$( ... )+`, `$( ... )?`, `$( ... ),*` 형태를 지원한다.
4. 템플릿 반복 확장:
`$( ... )* + ?`를 지원한다.
5. 안전 규칙:
반복 본문이 진전 없는(empty-progress) 형태면 `MacroRepeatEmpty`로 거부한다.
6. variadic 길이 불일치나 반복 문맥 외 variadic 사용은 전용 진단으로 거부한다.

## 29.4 스코프/선언 정책

1. 매크로 선언 허용 위치:
최상위, `nest` 본문.
2. 일반 블록 내 `macro` 선언은 금지한다.
3. item 문맥 매크로 호출은 `$foo(...);`처럼 세미콜론이 필요하다.
4. 이름 해석은 스코프 우선이며, 같은 스코프 전방참조를 허용한다.

## 29.5 위생성

기본 정책은 hygienic이며 binder 중심으로 적용한다.

1. 자동 gensym 대상 binder:
`let/set` 바인더, 함수 파라미터, `loop (x in ...)` 바인더.
2. 캡처 토큰은 rename하지 않는다(call-site 의미 보존).
3. 비위생 escape는 기본 금지다.

## 29.6 확장 예산과 결정성

`ExpansionBudget { max_depth, max_steps, max_output_tokens }`

1. AOT 기본값: `64 / 20000 / 200000`
2. JIT(parusd) 기본값: `32 / 8000 / 80000`
3. 하드 상한:
`max_depth <= 256`, `max_steps <= 200000`, `max_output_tokens <= 1000000`.
4. 사용자 입력은 clamp하며 경고를 출력한다.
5. 동일 입력/옵션에서 확장 결과는 결정적이어야 한다.

## 29.7 CLI/LSP 설정

`parusc`:

1. `-fmacro-max-depth=<N>`
2. `-fmacro-max-steps=<N>`
3. `-fmacro-max-output-tokens=<N>`

`parusd initializationOptions`:

1. `parus.macroBudget.maxDepth`
2. `parus.macroBudget.maxSteps`
3. `parus.macroBudget.maxOutputTokens`

## 29.8 `require(...)`와 매크로

1. `proto` 제약은 더 이상 `with require(expr)`를 사용하지 않으며, 본문 `require/provide` item을 사용한다.
2. `proto`의 `require foo(...) -> T;`는 함수 시그니처 항목이며 `def` 키워드를 허용하지 않는다.
3. 함수/블록 문맥의 `require(expr);` 단정 문장은 기존처럼 매크로 확장 결과를 포함한 bool 식을 대상으로 동작한다.
4. 내장 프레디킷 매크로(`$is_plain_data(Self)` 등)는 현재 코어에 도입하지 않았다.
5. `$[]`는 `$[If(...)]` 또는 `$[InstName(...)]` 형태의 compiler directive 호출만 허용한다.
6. `$[if(...)]` 레거시 문법은 제거되었으며 매크로로 우회할 수 없다.
7. 사용자 확장은 `inst` 선언을 사용하고, proc-macro/derive 스타일 코드생성 채널은 도입하지 않는다.
