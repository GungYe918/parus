# 01. Goals And Non-Goals

## 목표

1. LEI를 범용 빌드 오케스트레이션 시스템으로 고정한다.
2. 빌드 플랜 결합을 결정적이고 재현 가능하게 만든다.
3. 선언적 구성(`plan`, `import`, `export plan`)을 중심으로 빌드 입력을 기술한다.
4. 단일 번들부터 다중 번들 프로젝트까지 동일한 합성 규칙(`&`)으로 확장한다.
5. 제품 표면은 단일 LEI 시스템으로 제공하고, 내부는 Core/API/Profile 계층으로 분리한다.

## 비목표

1. 범용 스크립트 언어로 진화하지 않는다.
2. 동적/부작용 중심 자동화 언어가 되지 않는다.
3. 런타임 IO 언어로 확장하지 않는다.
4. 특정 호스트 정책(예: parus의 `master`)을 LEI Core에 하드코딩하지 않는다.

## 핵심 방향

1. 들여쓰기 의미를 쓰지 않는다.
2. 블록은 `{}`를 사용한다.
3. 배열은 `[]`를 사용한다.
4. 객체는 `.`로, 배열은 `[]`로 접근한다.
5. import는 별칭 기반 네임스페이스 접근(`alias::symbol`)으로 고정한다.

## 안정성 방향

1. `let`은 불변이다.
2. `var`는 가변이다.
3. `&`는 덮어쓰기가 아니라 충돌 감지 합성이다.
4. 실패는 침묵 fallback 없이 결정적 진단으로 종료한다.
